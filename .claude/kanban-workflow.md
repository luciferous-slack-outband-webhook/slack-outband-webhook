# Kanban ワークフロー詳細手順書

## 命名規則

- ファイル名: `{xxxx}_{title}.md`
- `xxxx`: 4桁の0パディング連番（例: `0001`, `0002`）
- `title`: 作業タイトル（スペースなし、ハイフン区切り推奨）
- `kanban/` と `logs/` で同じファイル名を使用する

## タイムスタンプ

JSTタイムゾーンの ISO 8601 形式を使用する。

```bash
TZ=Asia/Tokyo date +"%Y-%m-%dT%H:%M:%S+09:00"
```

出力例: `2026-04-11T21:30:00+09:00`

## kanban ファイルへの追記テンプレート

kanban ファイルへの追記は以下の構造で行う。タスク内容はユーザーが記述し、以降の セクションを Claude が追記する。

```markdown
## プラン

（プランモードで作成した実装計画をここに記載）

- 実施内容の概要
- 変更するファイル・コンポーネント
- 採用したアプローチと理由

---

## 完了サマリー

- **完了日時**: 2026-04-11T21:30:00+09:00
- **対応内容**:
  - 実施した変更の箇条書き
- **変更ファイル**:
  - `path/to/file1.rs`
  - `path/to/file2.rs`
- **備考**: （任意）
```

## ログファイルテンプレート

`logs/{xxxx}_{title}.md` に以下の構造で記録する。

```markdown
# {xxxx}_{title} 作業ログ

## 基本情報

- **タスクファイル**: kanban/{xxxx}_{title}.md
- **開始日時**: 2026-04-11T21:00:00+09:00
- **完了日時**: 2026-04-11T21:30:00+09:00

## 会話内容

（ユーザーの指示とClaudeの応答を時系列で記録）

### [21:00] ユーザー指示
（指示内容）

### [21:05] Claude 対応
（応答・実施内容）

## 編集したファイル

| ファイル | 変更内容 |
|---------|---------|
| `path/to/file.rs` | 変更の概要 |

## 実行したコマンド

```bash
cargo build
cargo test
```

## 判断・意思決定

- （設計判断やトレードオフの記録）

## エラー・問題

- （発生した問題と解決方法）
```

## 段階的ログ記録ルール

**重要**: ログは作業完了後にまとめて書くのではなく、段階的に追記すること。

1. **タスク開始時（最初のステップ）**: ログファイルのスケルトンを作成する
   - ヘッダー、基本情報（開始時刻）、各セクションの空テンプレートを書く
   - 完了日時は TBD としておく

2. **作業中（各ステップ完了時）**: ログファイルへ追記する
   - ファイルを編集したら「編集したファイル」セクションに追記
   - コマンドを実行したら「実行したコマンド」セクションに追記
   - 重要な判断をしたら「判断・意思決定」セクションに追記
   - エラーが発生したら「エラー・問題」セクションに追記

3. **作業完了時**: 最終化する
   - ログファイルの完了日時を更新する
   - 「会話内容」セクションに主要なやり取りをまとめる
   - kanban ファイルへ完了サマリーを追記する

## タスク検出ロジック

- 未完了タスク: `kanban/` 内の `.md` ファイルで `## 完了サマリー` を含まないもの
- `/kanban` コマンドに引数がない場合: 未完了タスクのうち番号が最大のものを選択

## 自動コミット手順

`/kanban` はタスク完了時（ログ最終化 + 完了サマリー追記のあと）に以下の手順で自動コミットする。

### ステージ対象の選別ルール

- 必ず含める:
  - `kanban/{xxxx}_{title}.md`
  - `logs/{xxxx}_{title}.md`
- 任意で含める: 完了サマリーの「変更ファイル」に列挙したソースファイル
- **含めない**: 今回のタスクに関係ない dirty file。`git add -A` / `git add .` は使用禁止。必ずパスを列挙する。

### 自動コミットメッセージテンプレート

```
[kanban {xxxx}] {kanban ファイルの H1 タイトル}

{要望を 1〜2 文に要約したもの}

## 対応内容
- {完了サマリーの「対応内容」から主要な箇条書きを転記}

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

- Subject は70文字以内。`{xxxx}` はタスク番号（例: `0002`）。
- 本文は HEREDOC で渡す（`git commit -m "$(cat <<'EOF' ... EOF)"`）。

### サンドボックスと git

`.git/**` への書き込みはサンドボックス制約に抵触する場合があるため、`git add` と `git commit` は Bash tool の `dangerouslyDisableSandbox: true` を指定して実行する。過去の `0000_cannot_git_pull` タスクでも同様に bypass している。

### コミット後

- `git log -1 --stat` で、意図したファイルがコミットに含まれていることを確認する
- worktree がクリーンである必要はない（`/kanban` と無関係な dirty file はそのまま残して構わない）
- push は `/kanban` では行わない（ユーザーが明示的に要求した場合のみ別途実施）
